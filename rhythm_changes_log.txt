OVERVIEW
--------
Over this session we built a full-featured jazz piano accompaniment and
improvisation program in Python, using the rtmidi library to send MIDI output
to a connected hardware or software instrument. The program grew step-by-step
from a bare-bones note player into a multi-threaded application with real-time
mood control.

The final program is rhythm_changes.py.

--------------------------------------------------------------------------------
STEP 1 — Initial script (rhythm_changes.py)
--------------------------------------------------------------------------------
Starting point: the workspace already contained random_midi_notes.py, which
demonstrated basic rtmidi usage (port enumeration, note-on/off, channel
selection).

Built from scratch: a stride-style piano accompaniment over Bb Rhythm Changes
(AABA, 32 bars).

Features introduced:
- Port discovery and selection on startup.
- MIDI program-change (GM Acoustic Grand Piano).
- Hardcoded "configuration section" constants at the top of the file:
    BPM, LOOPS, PORT, CHANNEL, PROGRAM, VEL_BASS, VEL_CHORD, NOTE_FILL
- Chord voicing model: each chord = (bass_pitch, [chord_tone_pitches]).
  Bass (octave 2) plays on beats 1 & 3; chord voicing (octave 3–4) on beats 2 & 4.
- Full Rhythm Changes progression in Bb:
    A-section : Bbmaj7 · Gm7 · Cm7 · F7 · Bb7 · Ebmaj7 · Edim7 · Bb/F
    Bridge    : D7 (2 bars) · G7 (2 bars) · C7 (2 bars) · F7 (2 bars)
- A "final chorus" variant that lands on the tonic rather than a turnaround.
- Graceful Ctrl+C handling with MIDI All-Notes-Off cleanup.

STEP 3 — --key parameter (transposition)
--------------------------------------------------------------------------------
Added `--key KEY` to transpose the entire progression into any of the 17
accepted key names: C, C#/Db, D, D#/Eb, E, F, F#/Gb, G, G#/Ab, A, A#/Bb, B.

Implementation:
- KEY_SEMITONES dict maps every name to a pitch-class (C=0 … B=11).
- TRANSPOSE global = signed semitone offset from the progression's natural key.
  Offset is folded to the range –6 … +6 so the shorter interval is always used
  (e.g. --key G on Rhythm Changes = –3, not +9).
- transpose_chord() helper shifts both the bass pitch and all chord-tone
  pitches by the same amount.
- build_chorus() applies transposition lazily if TRANSPOSE ≠ 0.

Example:
  python rhythm_changes.py --key F# --bpm 140

STEP 6 — Mood-driven real-time melody + PureData net control
--------------------------------------------------------------------------------
A full second thread generates an improvisatory melody on top of the
accompaniment. The "emotional character" of the melody is controlled by a
mood parameter that can be changed at any time while the program is running
by sending a TCP message from PureData (or any other tool).

New CLI parameters:
  --mood            happiness | sadness | anger | fear |
                    surprise | disgust | contempt | neutral  (default: happiness)
  --melody-channel  MIDI channel for melody, 0–15       (default: 1 = ch 2)
  --melody-program  GM program for melody, 0–127        (default: 66 = Tenor Sax)
  --net-port        TCP port for PureData netsend       (default: 3000)

Mood parameters (MOOD_PARAMS dict):
  Each mood is a dict controlling:
    octave_lo / octave_hi  — melodic register
    density                — probability of playing a note (vs rest) each step
    scale                  — 'major', 'minor', or 'chromatic'
    chord_w / scale_w / tension_w — weighted note-pool composition
    vel_base / vel_spread  — velocity mean and random spread
    step_beats             — rhythmic step size (0.25 = ♬, 0.5 = ♪, 1.0 = ♩, etc.)
    fill                   — note-on fraction (low = staccato, high = legato)

Mood characters:
  happiness  — bright, dense eighth notes, chord/scale tones, high velocity
  sadness    — low register, slow, minor scale, very legato, soft
  anger      — fast sixteenth notes, chromatic, very loud, high tension
  fear       — sparse, wide velocity swings, chromatic, low register
  surprise   — very wide register (oct 3–7), erratic velocity
  disgust    — heavy tension notes, medium-slow, detached
  contempt   — minimal and staccato, nearly flat dynamics, sparse
  neutral    — mid register (oct 4–5), moderate density dotted-eighth steps,
               mostly chord/scale tones, gentle velocity, nearly no tension

Melody generation (_melody_candidates):
  Builds a weighted pool from: chord tones · scale tones · tension notes.
  Weights derived from chord_w / scale_w / tension_w for the current mood.
  Scales used: major [0,2,4,5,7,9,11], minor [0,2,3,5,7,8,10], chromatic (all 12).

Threads:
  melody_thread_func    — loops over melody steps; reads _current_chord and
                          MOOD; generates notes on MELODY_CHANNEL.
  network_listener_func — TCP server bound on NET_PORT; accepts PureData netsend
                          connections, parses mood-name tokens (delimited by
                          ;, \n, \r, \0), validates against _MOOD_ALIASES,
                          updates the MOOD global thread-safely under _mood_lock.

Shared state:
  _current_chord  — updated by play_chorus() before every chord slot
  _mood_lock      — threading.Lock() protecting MOOD + _current_chord reads
  _stop_event     — threading.Event() used to signal all threads to exit

Cleanup: both MIDI channels receive CC 123 (All Notes Off) on exit.

PureData usage (example patch fragment):
  [symbol happiness( → [netsend 127.0.0.1 3000]
  [symbol anger(    → [netsend]
  Accepted aliases: happy, sad, angry, fearful, surprised, disgusted.

STEP 8 — --bass-variation parameter
--------------------------------------------------------------------------------
Added randomised bass-line variation controlled by a single probability knob.

New CLI parameter:
  --bass-variation F   float 0.0–1.0 (default: 0.0 = no variation)

On every bass-root beat, a random check fires with probability equal to the
parameter value. If it passes, one of two variation types is chosen at random:

  1. Chromatic passing note (subdivision):
     The beat is split into two equal eighth notes.
     First eighth : the main bass root at full velocity.
     Second eighth: one semitone up or down (chosen randomly), played ~18
                    velocity units softer — a ghost passing tone.
     Total duration is unchanged; timing stays locked.

  2. Alternate chord tone (substitution):
     The root is replaced for the whole beat by an idiomatic alternate:
       stride style       → perfect fifth (root + 7 semitones)
       boogie-woogie      → major sixth  (root + 9 semitones)
     These are the most natural bass substitutions in each style.

Beats that are not bass-root beats (the chord voicing on beat 2/4 in stride,
or the fifth + chord stab on beat 2 in boogie-woogie) are never affected.

Implementation:
  _play_bass_note(midiout, pitch, beat, vel, *, alt=None)
    Internal helper that encapsulates the variation logic.
    Called from play_slot_stride() and play_slot_boogie() in place of the
    previous inline note_on / time.sleep / note_off sequence.
  BASS_VARIATION global set from args.bass_variation in main().

Examples:
  python rhythm_changes.py --bass-variation 0.3
  python rhythm_changes.py --style boogie-woogie --bass-variation 0.5

STEP 9 — --mood-tempo parameter
--------------------------------------------------------------------------------
The mood now optionally influences the playback tempo as well as the melody
character. A new scalar parameter controls how strongly the mood bends the BPM.

New CLI parameter:
  --mood-tempo F   float 0.0–1.0 (default: 0.0 = no tempo effect)

Per-mood BPM multipliers (_MOOD_BPM_FACTORS) at MOOD_TEMPO = 1.0:
  anger      ×1.18   driven, fast
  surprise   ×1.10   sudden burst
  happiness  ×1.08   bright, energetic
  fear       ×1.06   nervous energy
  neutral    ×1.00   baseline — no change
  contempt   ×0.93   detached, unhurried
  disgust    ×0.88   reluctant, dragging
  sadness    ×0.80   heavy, slow

At intermediate values the multiplier is interpolated linearly between 1.0
(no effect) and the full factor. For example, --mood-tempo 0.5 with anger
gives an effective multiplier of ×1.09 (half of the ×1.18 deviation from 1.0).

Implementation:
  _mood_beat(base_beat) — helper that returns the mood-adjusted quarter-beat
    duration. Called at the start of play_slot_stride(), play_slot_boogie(),
    and each melody step in melody_thread_func(). The global BPM is never
    mutated; the scaling lives entirely in the timing arithmetic.
  MOOD_TEMPO global set from args.mood_tempo in main().

When the mood changes at runtime (via TCP), the new tempo takes effect
immediately on the next beat. The startup header prints the resolved BPM
when it differs from the base value (e.g. “130 BPM → 153 BPM (anger)”).
The [net] listener also logs the effective BPM on each mood change.

Examples:
  python rhythm_changes.py --mood anger --mood-tempo 1.0
  python rhythm_changes.py --mood sadness --mood-tempo 0.5 --bpm 140

STEP 10 — Face-gated playback (mood_detector integration)
--------------------------------------------------------------------------------
Playback now starts and stops automatically depending on whether mood_detector
has a face in frame, coordinated through the same TCP connection used for mood
messages.

New TCP tokens (received by rhythm_changes.py on --net-port):
  face_on    — a face has been confirmed present; resume playing
  face_off   — no face confirmed; pause and silence both MIDI channels

Behavior in rhythm_changes.py:
  _play_event (threading.Event, starts set) controls playback.
  play_chorus() checks _play_event before every chord slot:
    • On pause: sends All Notes Off (CC 123) on both accomp. and melody
      channels, then blocks until the event is set again.
    • On resume: continues from the exact chord slot it paused on.
  melody_thread_func() releases any held melody note and waits when
    _play_event is cleared.
  Network listener handles 'face_on' / 'face_off' alongside mood tokens.

Backward compatibility:
  Running rhythm_changes.py alone: _play_event starts set, so playback
  begins immediately as before. Face-gating only activates once
  mood_detector.py connects and sends its initial 'face_off'.

STEP 11 — Melodic improvements: stepwise motion, arc contour, phrase planning
--------------------------------------------------------------------------------
Date: April 29, 2026

Three interrelated improvements to make the melody lines sound more musical
and less like random note selection.

1. Stepwise / proximate motion weighting
   Previously each melody note was chosen with uniform probability from the
   candidate pool, resulting in frequent large leaps (octave+) with no regard
   for the previous pitch.  Now the pool is sampled with distance-based weights:

     weight = max(1, 12 − |candidate − prev_note|)

   A note 1 semitone away gets weight 11; a note an octave away gets weight 1.
   Large leaps are still possible but ~11× less likely than a half-step.
   The first note of a phrase (prev_note is None) still uses uniform choice.

2. Melodic direction / contour state
   A new _build_phrase() helper pre-plans each phrase with an arc-shaped
   contour.  At phrase start, a direction is chosen:
     • ascending-then-descending (60% probability — common in bop lines)
     • descending-then-ascending (40%)
   A peak position is placed randomly in the middle third-to-two-thirds of
   the phrase so arcs vary.  Each step blends two weights:
     • proximity  : max(1, 7 − |candidate − current|)  — prefer stepwise
     • direction  : ×3 boost for notes moving the intended way
   The combined weight naturally steps toward the peak, then steps back down
   (or vice-versa), producing the arch shapes characteristic of jazz lines.

3. Phrase-level planning (4–8 note phrases)
   melody_thread_func() now works through a pre-planned phrase list rather
   than deciding each note independently:
     • When phrase_idx ≥ len(phrase), a new phrase of 4–8 notes is assembled
       via _build_phrase() before playback begins.
     • On a rest step (density check fails) the current phrase is discarded,
       so the next played note starts a fresh phrase — creating natural breath
       points between melodic ideas.
     • On face-off pause the phrase is also cleared, so resumption always
       begins a new idea rather than continuing a stale arc.

New function: _build_phrase(chord_data, params, prev_note, phrase_len) → list[int]
  Calls _melody_candidates() once to build the candidate pool, then constructs
  the phrase note-by-note using the combined proximity+direction weight scheme.

New state in melody_thread_func():
  phrase      list[int]  — the current pre-planned phrase
  phrase_idx  int        — index of the next note to play within phrase
  prev_note   int | None — last played pitch, seeded into the next phrase entry

STEP 12 — Melodic improvements: approach notes, motif repetition,
          rhythmic grouping, leap clamping
--------------------------------------------------------------------------------
Date: April 29, 2026

Five further improvements building on Step 11 to make the melody sound like
a real improvised jazz line rather than a random walk over the changes.

4. Approach notes on chord changes
   A new _approach_note(chord_data, prev_note) helper returns the pitch
   1 semitone below the nearest chord tone of the incoming chord to prev_note,
   kept within an octave of prev_note.  Whenever the accompaniment moves to a
   new chord, the phrase is immediately refreshed and this approach pitch is
   used to pin the phrase's first note — the classic bebop "lean-into-the-
   change" effect.

5. Motivic repetition / recall
   When a phrase is fully played through, it is saved as `motif`.  On the next
   phrase build there is a 25 % chance to replay it, transposed so its opening
   note lands as close as possible to prev_note.  If the transposition takes
   any note outside the current candidate-pool range the fallback is a freshly
   built phrase, preventing harmonically out-of-range repetition.

6. Rhythmic grouping / phrase breathing
   Two counters collaborate to turn scattered random rests into clean blocks:
     consec_rests — increments each time the density check votes for a rest.
     rest_steps   — when consec_rests reaches 2, set to randint(1, 3) and
                    consec_rests resets.
   While rest_steps > 0 each loop iteration simply waits one step and
   decrements the counter, forcing a silence window of 1–3 continuous steps
   before the melody resumes.

7. Target-note resolution on chord changes (combined with #4)
   The approach_target computed at chord change time is passed through to
   _build_phrase() as a keyword argument.  Inside _build_phrase() the first
   note is snapped via min(pool, key=lambda p: |p − approach_target|) so the
   phrase always opens on or near the approach pitch, landing on a chord tone
   after one step.

8. Leap-width clamp on fast sub-beat steps
   Inside _build_phrase(), when step_beats < 0.75 (sixteenth / eighth note
   speed) the candidate pool at each step is filtered to notes within ±12
   semitones of the current pitch.  An `or pool` fallback ensures the pool
   never becomes empty in a very narrow register.

New function: _approach_note(chord_data, prev_note) → int

Updated function: _build_phrase(chord_data, params, prev_note, phrase_len, *,
                                 step_beats=1.0, approach_target=None) → list[int]
  Added step_beats and approach_target keyword parameters.

New state in melody_thread_func():
  prev_chord_data  — previous chord slot; compared each step to detect change
  motif            list[int]  — last fully played phrase, used for recall
  rest_steps       int        — steps remaining in forced rest window
  consec_rests     int        — consecutive rest-vote counter

STEP 13 — Mood-driven dissonance in accompaniment chords + melody tension boost
--------------------------------------------------------------------------------
Date: April 29, 2026

For moods that call for it, the accompaniment chord voicing now randomly
includes an extra chromatic "colour" tension note, and the melody note-pool
weights for those moods were adjusted to include a higher proportion of
tension tones.

New MOOD_PARAMS key: accomp_dissonance (float 0.0–1.0)
  Probability that a colour/tension note is added to the chord stab on any
  given beat.  Values per mood:
    happiness  0.00  — clean, diatonic
    sadness    0.10  — occasional wistful blue note
    anger      0.50  — half of all chord beats clashing
    fear       0.45  — unsettling crunch
    surprise   0.20  — unexpected harmonic twist
    disgust    0.55  — heaviest dissonance (most chromatically abrasive)
    contempt   0.00  — dry and empty, no tension
    neutral    0.00  — plain, no colouring

New constant: _COLOUR_INTERVALS = [1, 3, 6, 8]
  Semitone offsets from the bass root corresponding to the classic upper-
  structure tensions used in jazz:
    1  = b9   (minor ninth)
    3  = #9   (augmented ninth / Hendrix chord)
    6  = b5 / #11  (tritone)
    8  = b13  (minor thirteenth)

New helper: _colour_tone(bass, tones) → int | None
  Chooses one interval randomly, maps it to a pitch class above the bass
  root, then finds the closest occurrence within the octave range spanned
  by the existing chord tones (lo … lo+12).  Returns None if no candidate
  fits (e.g., the interval lands exactly on an existing tone's octave limit).

Accompaniment changes (play_slot_stride, play_slot_boogie):
  On the chord-stab beat (beat 2/4 in stride; beat 2 in boogie), _colour_tone
  is called if a random draw < accomp_dissonance.  The colour note is struck
  alongside the normal voicing at VEL_CHORD − 14 (soft enough to colour
  without dominating the voicing) and note-off'd with the rest of the chord.
  Bass beats are never affected.

Melody tension_w adjustments for affected moods:
  sadness   tension_w 0.08 → 0.12, scale_w 0.54 → 0.50
             (more minor-scale colour notes for a bluesier feel)
  surprise  tension_w 0.25 → 0.30, scale_w 0.30 → 0.25
             (more unexpected upper-structure notes in the melody)

STEP 14 — Melody velocity boost for soft moods + accompaniment humanisation
--------------------------------------------------------------------------------
Date: April 29, 2026

Two mixing fixes to make the melody audible and the piano feel more human.

1. Melody velocity boost (sadness, disgust, neutral)
   These three moods had vel_base values at or below VEL_CHORD (64), causing
   the melody to be buried under the chord voicing.  Updated values:

     sadness  vel_base 52 → 74, vel_spread 12 → 14   range: 60–88
     disgust  vel_base 58 → 76, vel_spread 10 → 12   range: 64–88
     neutral  vel_base 68 → 80, vel_spread 10 → 12   range: 68–92

   All three remain considerably softer than happiness (90) and anger (108),
   preserving emotional contrast while ensuring the melody can be heard.

2. Accompaniment velocity humanisation
   Every note_on call in the piano accompaniment now passes through a new
   helper _hv() instead of using the raw VEL_BASS / VEL_CHORD constants:

     _hv(vel, spread=7) → max(1, min(127, vel + randint(-spread, spread)))

   Each note receives its own independent ±7 offset, so:
     • Bass notes, chord tones and colour tones all vary independently.
     • Inner chord voices land with slightly unequal weight, as a real
       pianist's fingers would.
     • The overall dynamic character (loud/soft per mood) is unchanged
       because the offsets are symmetric around the target velocity.

   Applied in: play_slot_stride() and play_slot_boogie() — all note_on
   calls for bass, chord tones, fifth (boogie), and colour tones.

STEP 15 — --melody-vel-boost and --melody-octave parameters
--------------------------------------------------------------------------------
Date: April 29, 2026

Two new CLI parameters to give fine-grained control over the melody's
overall level and register independently of the per-mood presets.

--melody-vel-boost N   integer, default 0, range −127–127
  A fixed offset added to every melody note's computed velocity after the
  per-mood vel_base ± vel_spread calculation:
    vel = clamp(1, 127, vel_base + randint(-spread, spread) + MELODY_VEL_BOOST)
  Use positive values to lift the melody above the accompaniment mix;
  negative values to push it further back.  The final result is always
  clamped to the valid MIDI range 1–127.

--melody-octave N   integer, choices −2 −1 0 1 2, default 0
  Transposes every played melody note by N octaves (N × 12 semitones) at
  playback time.  Applied to the output note only; phrase planning, arc
  contour, motif storage and approach-note logic all operate in the
  original register so musical structure is fully preserved.
    note_out = clamp(1, 127, phrase_note + MELODY_OCTAVE * 12)

Both constants follow the existing DEFAULT_* / global pattern and are
assigned from args in main().

Examples:
  python rhythm_changes.py --melody-vel-boost 20
  python rhythm_changes.py --melody-octave 1
  python rhythm_changes.py --mood sadness --melody-vel-boost 15 --melody-octave 1

STEP 17 — --changes rotate + --rotate-every: cycling through all three changes
--------------------------------------------------------------------------------
Date: May 8, 2026

Added a fourth value for --changes that automatically cycles through Blues,
Rhythm Changes, and Coltrane Changes in sequence, keeping the tonality
consistent across all three.

New CLI parameter:
  --changes rotate (alias: ro)
    Plays Blues → Rhythm Changes → Coltrane Changes repeatedly, rotating
    after a configurable number of choruses.  Each changes type is played in
    full-length choruses (12, 32, and 16 bars respectively).

New CLI parameter:
  --rotate-every N   integer ≥ 1, default 4
    Number of choruses of each changes type to play before rotating to the
    next.  Applies equally to all three.  Examples:
      --rotate-every 4  → 4 Blues, 4 Rhythm, 4 Coltrane, 4 Blues, …
      --rotate-every 1  → one chorus of each in turn

Tonality / transposition in rotate mode:
  Blues (ref key Bb) and Rhythm Changes (ref key Bb) share the same natural
  key.  Coltrane Changes have a natural key of B, one semitone higher.
  In rotate mode a single target key is fixed at startup (default Bb, or
  whatever --key is passed).  When the rotation switches to Coltrane Changes,
  TRANSPOSE is recomputed relative to B so the progression sounds in the same
  key as the other two changes types — effectively transposing Coltrane −1
  semitone into Bb.  The console prints the change:
    [rotate] → Coltrane Changes  (key: Bb, transpose: -1 st)

Implementation:
  _ROTATE_SEQUENCE = ['blues', 'rhythm', 'coltrane']   (module-level constant)
    Defines the fixed rotation order.

  In main(), when args.changes == 'rotate':
    • _rotate_mode flag is set True.
    • CHANGES is initialised to the first item in _ROTATE_SEQUENCE.
    • _rotate_target_key is stored (the fixed tonality throughout the session).
    • TRANSPOSE is computed for the first changes type as normal.

  In the main playback loop, before each chorus:
    segment_idx = ((chorus - 1) // ROTATE_EVERY) % len(_ROTATE_SEQUENCE)
    When the computed segment_idx maps to a different changes type than the
    current CHANGES, CHANGES and TRANSPOSE are updated atomically, and the
    transition is logged to the console.

  The `is_final` logic (--loops) continues to work as before: the final
  chorus tag is applied to whichever changes type happens to be active.

New constant: DEFAULT_ROTATE_EVERY = 4
New global:   ROTATE_EVERY (set from args.rotate_every in main())

Startup header when --changes rotate:
  "Rotate (Blues → Rhythm → Coltrane, N choruses each) — Bb — …"

Examples:
  python rhythm_changes.py --changes rotate
  python rhythm_changes.py --changes ro --rotate-every 2
  python rhythm_changes.py --changes ro --key F --rotate-every 3

STEP 18 — Piano keepalive during no-face pause
--------------------------------------------------------------------------------
Date: May 10, 2026
File: rhythm_changes.py

When mood_detector signals 'face_off' the accompaniment and melody threads
both pause and produce no MIDI output.  Some digital pianos enter an
automatic sleep/power-saving mode after a period of silence, which causes
the instrument to stop responding until it is woken.

Fix: a new dedicated daemon thread, _keepalive_thread_func(), fires once
every 60 seconds while _play_event is cleared (no-face / paused mode) and
sends a MIDI note-on for middle C (pitch 60) at velocity 0 on the
accompaniment channel.  Velocity 0 is defined by the MIDI standard as
equivalent to a note-off and produces no audible sound; it is sufficient
to reset the piano's inactivity timer.

Implementation details:
  _keepalive_thread_func(midiout, interval=60.0)
    • Blocks on _play_event.wait() while music is playing (efficient, no
      busy loop).
    • Once paused, enters a secondary inner loop that fires the keepalive
      message and then waits in 1-second increments so it reacts quickly
      to both stop (_stop_event) and resume (_play_event) signals.
    • On resume it exits the inner loop and returns to blocking on
      _play_event.wait() until the next pause.
    • Thread is daemon=True so it does not prevent clean shutdown.

  In main(): a keepalive_thread is created and started alongside net_thread
  and mel_thread, using the same midiout instance (accompaniment port).

  Console output: prints "[keepalive] Sent silent note to prevent piano
  sleep" each time a keepalive is fired, for monitoring.

STEP 19 — --demo and --keepalive-interval parameters
--------------------------------------------------------------------------------
Date: May 11, 2026
File: rhythm_changes.py

Two new CLI parameters for unattended / exhibition use.

--demo N   float, default 0 (disabled)
  When set to a positive number, a new daemon thread (_demo_thread_func)
  changes the active mood to a random different mood every N seconds.
  The thread runs only while mood_detector has not yet connected: the first
  'face_off' token received over TCP sets a new _face_gated threading.Event,
  which the demo thread monitors and immediately exits upon.  From that point
  mood control is fully handed over to mood_detector as normal.

  Implementation details:
    _face_gated  (threading.Event) — set the first time 'face_off' arrives
      from the network listener; replaces the previous unconditional clear.
    _demo_thread_func(interval) — loops with 0.5 s increments watching both
      _stop_event and _face_gated; on each full interval picks a mood from
      MOOD_PARAMS that differs from the current one and updates MOOD under
      _mood_lock. Prints "[demo] Mood → <name>" on each change.
    On startup (when DEMO > 0) prints:
      "[demo] Active — mood will change every N s until mood_detector connects."

  Example:
    python rhythm_changes.py --demo 10 --changes rotate

--keepalive-interval N   float, default 300
  Sets the number of seconds between silent keepalive MIDI notes sent to the
  piano while in no-face / paused mode (previously hard-coded to 60 s).
  The value is passed directly to _keepalive_thread_func as its interval
  argument.  Set to 0 to effectively disable keepalives (the thread enters
  an infinite wait without ever firing).

  New constant : DEFAULT_KEEPALIVE = 300
  New global   : KEEPALIVE  (set from args.keepalive_interval in main())

  Example:
    python rhythm_changes.py --keepalive-interval 120
